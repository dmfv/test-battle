# Архитектурный подход решения

В проекте выбран максимально расширяемый вариант архитектуры: `Core` предоставляет абстракции мира, действий, эффектов, реакций и жизненного цикла, а `Features` собирают новых юнитов из поведения, набора характеристик и реактивных трейтов.

Главные цели этого варианта:
- не заставлять `Core` знать про конкретные типы юнитов;
- позволить добавлять новые механики через новые `Action`, `Trait`, `Effect` и `Factory`;
- не раздувать один большой `TurnContext` или базовый класс `Unit`;
- разделить "выбор действия", "применение эффекта" и "побочные реакции".

## Слои

- `App` склеивает `IO`, `Core` и `Features`.
- `IO` отвечает только за парсинг сценария и печать событий в требуемом формате.
- `Core` владеет состоянием мира и общими правилами симуляции.
- `Features` описывает конкретные игровые сущности без правок в ядре.

## Доменные принципы

### 1. Юнит не наследуется, а собирается

Юнит моделируется как агрегат данных:
- identity: `id`, `typeName`, `creationOrder`;
- spatial state: позиция и активный марш;
- optional health: здоровье не является обязательным для каждого объекта;
- capabilities: может ли двигаться, занимает ли клетку, можно ли атаковать в ближнем/дальнем бою, сколько шагов делает за ход;
- attributes: произвольные числовые характеристики по именам;
- active behavior: выбор действия в свой ход;
- traits: реактивные способности, которые отвечают на события мира.

Такой подход нужен, чтобы без специальных `if` поддерживать будущие сущности вроде `Crow`, `Mine`, неубиваемые объекты и объекты без HP.

### 2. Поведение возвращает `Action`, а не применяет мир напрямую

Активное поведение юнита только принимает решение. Оно не должно:
- наносить урон напрямую;
- удалять юнита;
- логировать события;
- искать другие системы.

Вместо этого оно возвращает доменное намерение `Action`, например:
- `MoveAlongMarchAction`
- `MeleeAttackAction`
- `RangedAttackAction`
- `HealAction`
- `SelfDestructAction`

### 3. Системы не ищут друг друга

Связывание зависимостей происходит сверху вниз:
- `Simulation` владеет `World` и системами;
- `EffectResolver` оркестрирует применение действий и эффектов;
- конкретные системы получают зависимости через конструкторы и не используют service locator.

Это уменьшает связанность и делает код предсказуемым для сопровождения.

### 4. Реактивные способности живут в `Trait`

Часть механик неудобно держать только в активном ходе юнита. Например:
- "взорваться, если HP стало мало";
- "отреагировать на начало хода";
- "породить действие после получения урона";
- "сработать при пометке на удаление".

Для этого в `Core` вводится отдельная ось расширения: `IUnitTrait`.

Трейт получает domain event и может вернуть новые `Action`, которые `Core` выполнит через тот же pipeline.

### 5. `Action -> Systems -> Events -> Traits`

Основной pipeline выглядит так:

1. `Behavior` выбирает `Action`.
2. `EffectResolver` направляет `Action` в нужную систему.
3. Система меняет `World`, выпускает domain events и внешние IO-friendly события.
4. Domain events рассылаются в `TraitEngine`.
5. Traits могут породить дополнительные `Action`.
6. Цепочка продолжается, пока очередь действий не опустеет.

Это позволяет добавлять механики расширением, а не переписыванием уже готовых юнитов.

## Жизненный цикл здоровья и смерти

HP вынесен в отдельную optional-компоненту, потому что:
- некоторые сущности могут не иметь HP;
- некоторые сущности могут получать урон, но не исчезать;
- некоторые сущности могут быть вообще неатакуемыми.

Смерть отделена от удаления:
- сначала применяются изменения здоровья;
- при переходе в терминальное состояние юнит помечается на удаление;
- реальное удаление выполняется отдельной системой жизненного цикла в согласованной точке тика.

## Расширение без правок `Core`

Обычное добавление нового юнита должно требовать:
- нового `Behavior`;
- опционально новых `Trait`;
- новой `Factory` в `Features`;
- регистрации фабрики в `App`.

`Core` должен меняться только если в игре появляется новая общая категория механики, а не просто новый юнит.
