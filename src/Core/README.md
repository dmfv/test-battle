`Core` — это расширяемая основа проекта.
- В будущем только вы несёте полную ответственность за этот код и контролируете все его изменения, предоставляя множеству разработчиков возможность создавать игровые функции в отдельной директории.
- Не станьте узким горлышком в разработке.

# Задача `Core`

`Core` должен предоставлять стабильную платформу для игровых механик и при этом не знать о конкретных игровых сущностях.

Он отвечает за:
- хранение мира и юнитов;
- инварианты карты и жизненного цикла;
- общий execution pipeline;
- общие системы боя, движения, лечения и удаления;
- доменные события и реактивные extension points;
- контракты для `Features`.

`Core` не должен:
- зависеть от `IO`;
- зависеть от конкретных `Swordsman`, `Hunter`, `Tower` и т.д.;
- принимать решения за конкретные боевые стили юнитов;
- обрастать методами "под каждую новую абилку".

# Основные абстракции

## `Unit`

Юнит в ядре не моделируется наследованием. Это агрегат, состоящий из:
- identity;
- optional health;
- capabilities;
- attributes;
- активного `Behavior`;
- набора реактивных `Traits`.

## `IUnitBehavior`

Определяет только активное действие юнита в его ход.

Контракт:
- получает read-only view мира;
- получает random engine;
- возвращает доменное `Action`.

`Behavior` не изменяет `World` напрямую.

## `IUnitTrait`

Определяет реакцию юнита на domain events.

Контракт:
- получает event;
- может вернуть zero-or-more `Action`;
- используется для пассивок, триггеров и отложенных механик.

Это ключевой extension point для механик наподобие:
- "взорваться при низком HP";
- "в начале хода проверить окружение";
- "реагировать на нанесенный урон";
- "сделать контратаку после получения события".

## `Action`

`Action` это намерение, выбранное поведением или трейтом.  
Ядро знает только общий набор видов действий, например:
- перемещение по активному маршу;
- атака ближнего боя;
- дальняя атака;
- лечение;
- самоуничтожение;
- отсутствие действия.

Если появляется новая общая категория механики, `Core` расширяется новым видом `Action` и системой применения.

## `Event`

В ядре есть отдельный слой domain events.

Они нужны для:
- реактивных механик;
- отделения внутренней логики от внешнего лога;
- тестирования и отладки pipeline.

Важно:
- domain event не равен IO event;
- `IO` может адаптировать domain event в нужный формат логирования.

# Execution pipeline

Один тик должен обрабатываться в следующих фазах:

1. `LifecycleSystem` удаляет юнитов, которые были помечены на удаление ранее.
2. `TurnEngine` обходит юнитов в порядке создания.
3. Для очередного юнита вызывается его `Behavior`.
4. `Behavior` возвращает `Action`.
5. `EffectResolver` передает `Action` нужной системе.
6. Система меняет `World` и публикует domain events.
7. `TraitEngine` рассылает события всем трейтам и собирает их ответные `Action`.
8. Очередь действий обрабатывается до исчерпания.

Такой подход предотвращает прямые связи между системами и упрощает добавление новых реактивных механик.

# Структура директорий `Core`

- `Actions/` доменные намерения и эффекты
- `Common/` базовые алиасы и общие типы
- `Contracts/` публичные интерфейсы ядра
- `Events/` внутренние события симуляции
- `Factories/` реестр фабрик юнитов
- `Model/` состояние мира и юнитов
- `Simulation/` оркестрация тиков и сценарных команд
- `Systems/` реализация общих игровых правил

# Правила расширения

Изменение в `Core` оправдано только если:
- появляется новая общая категория правила;
- появляется новый универсальный hook жизненного цикла;
- нужен новый тип доменного действия или события.

Изменение в `Core` не оправдано, если:
- добавляется еще один юнит с уже известными возможностями;
- меняются формулы уже существующего юнита;
- появляется новый набор статов, который можно положить в `attributes`;
- нужно добавить новый сценарный `SPAWN_*` в `IO` и новую `Factory` в `Features`.
