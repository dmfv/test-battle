`Feature` — это директория для реализации игровых фичей.
- Здесь разработчики размещают код юнитов и их поведение, используя возможности, предоставленные `Core`.

# Что должно жить в `Features`

Каждый тип юнита должен собираться здесь из готовых extension points ядра:
- `Behavior`
- `Trait`
- `Factory`
- feature-specific helpers

Ожидаемый состав фичи:
- `<UnitName>Behavior`
- `<UnitName>Factory`
- `<UnitName>Traits...` при необходимости

Примеры:
- `SwordsmanBehavior` выбирает melee-атаку или шаг по маршу;
- `HunterBehavior` выбирает ranged/melee/move;
- `LowHpExplodeTrait` реагирует на изменение здоровья;
- `MineProximityTrait` реагирует на начало хода.

# Что не должно жить в `Features`

- общие правила карты;
- общий расчет урона и лечения;
- порядок удаления юнитов;
- lifecycle тика;
- прямое управление чужими системами;
- формат ввода/вывода сценариев.

# Практика расширения

При добавлении нового юнита разработчик в `Features` должен:

1. выбрать или создать `Behavior`;
2. добавить необходимые `Trait`;
3. описать стартовые `Capabilities`;
4. положить стартовые числовые статы в `attributes`;
5. зарегистрировать `Factory` в сборке приложения.

Если для новой сущности требуется изменение `Core`, надо сперва проверить:
- это действительно новый класс механики, а не частный случай;
- нельзя ли выразить поведение уже существующими `Action`, `Trait` и `Capabilities`;
- не приведет ли изменение к знанию `Core` о конкретном юните.
